# Intake Question Flow

Use this when a product idea, feature request, or workflow change is still vague.

## Rule

Ask one question at a time. Continue only after the user's answer changes the product spec, open questions, or assumptions.

## Why

Large question lists create fatigue and make users answer shallowly. One focused question at a time produces better requirements and makes unclear answers easier to catch.

## Question Order

Prefer this order, but adapt to the user's answer:

1. User and usage context.
2. Primary job-to-be-done.
3. Current workaround or pain.
4. Success moment.
5. Must-have flow.
6. Non-goals.
7. Data inputs, outputs, and permissions.
8. Platform, integrations, and deployment.
9. Cost, privacy, safety, and reliability constraints.
10. Edge cases and recovery.
11. How the user will judge quality.

## Handling Answers

- If an answer is concrete, record it in `product-spec.md` or raw notes.
- If an answer is ambiguous, ask a narrower follow-up before planning.
- If the user wants speed, state assumptions explicitly and mark them as reversible.
- If the user corrects the questioning style, capture an evolution signal.

## Stop Condition

Stop intake when the product spec can support a design brief without guessing the core user, problem, scope, constraints, and success criteria.
